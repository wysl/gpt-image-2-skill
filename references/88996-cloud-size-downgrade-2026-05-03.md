# 88996.cloud 单端点高分辨率返回行为（2026-05-03）

## 结论

`88996.cloud` 在 GPT-Image 请求 `2160x3840` 时，**可能返回 HTTP 200 但静默降级尺寸**，至少本会话两次稳定返回：

- 请求：`2160x3840`
- 实际返回：`941x1672`
- 返回格式：`data[0].url`（不是 `b64_json`）

因此：
1. 不能只看 HTTP 200 判断“4K 成功”
2. 不能假设 `response_format=b64_json` 会被遵守
3. 必须下载实际文件再校验 PNG IHDR 宽高

## 复现方式

- 只打端点1（priority=1）
- 使用一次性 `requests.post()` 直连 `https://88996.cloud/v1/images/generations`
- prompt 可为保守英文人像/海报提示词
- payload 中设置：
  - `size=2160x3840`
  - `quality=high`
  - `response_format=b64_json`

## 实测现象

### 测试1
- prompt：较完整的保守古风海报 prompt
- HTTP：200
- 返回：`data[0].url`
- 下载后尺寸：`941x1672`

### 测试2
- prompt：更短、更保守版本
- HTTP：200
- 返回：`data[0].url`
- 下载后尺寸：`941x1672`

### 模板路径测试
- `poster-cosplay/run.py`
- 日志显示请求尺寸仍为 `2160x3840`
- 保存出的 PNG 实际尺寸依然是 `941x1672`

## 推断

更像是端点/provider 层面的静默缩放/内部上限，而不是：
- prompt 过长
- prompt 过于敏感
- 仅直调 API 才会发生

因为：
- 更短 prompt 仍然降级
- 模板调用也同样降级
- 返回成功且图像内容正常

## 对用户汇报的规范

必须明确分开写：
- 请求尺寸：`2160x3840`
- 实际返回尺寸：`941x1672`
- 使用端点：`88996.cloud`

不要把这种情况表述成“4K 生成成功”。

## 用户偏好（本会话确认）

- 白边若是 API/端点问题：**不裁边**，直接发原图
- **不额外生成 JPG 副本**
- 原图超过 Telegram 10MB 限制时，才考虑压缩派生文件
- 用户说“只试一次”时：
  - 只打指定端点
  - 不 fallback
  - 不多次重试
  - 不顺手生成额外版本
